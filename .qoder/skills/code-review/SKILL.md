# Code Review

## 评审流程

1. **理解上下文**：先了解代码变更的目的和背景
2. **检查正确性**：验证逻辑是否正确，是否处理了边界情况
3. **评估安全性**：检查潜在的安全漏洞
4. **评估可维护性**：代码是否清晰、易读、易测试
5. **提供反馈**：按优先级分类反馈

## 评审清单

### 功能性
- [ ] 代码逻辑正确，无 bug
- [ ] 边界条件和错误情况已处理
- [ ] 无死代码或无用代码

### 安全性
- [ ] 无 SQL 注入、XSS 等安全漏洞
- [ ] 敏感信息（密码、密钥）未硬编码
- [ ] 输入验证和输出编码正确

### 代码质量
- [ ] 命名清晰、有意义
- [ ] 函数/方法长度适中，职责单一
- [ ] 代码重复已消除或合理化
- [ ] 注释清晰，解释"为什么"而非"是什么"

### 测试
- [ ] 新增代码有相应的测试覆盖
- [ ] 测试用例覆盖正常和异常情况

### 性能
- [ ] 无明显性能问题
- [ ] 资源使用合理

## 反馈格式

按优先级分类：

- **🔴 必须修复**：阻止合并的严重问题（bug、安全漏洞）
- **🟡 建议改进**：影响代码质量，但可后续处理
- **🟢 可选优化**：锦上添花的小建议

## 示例反馈

```
🔴 必须修复
- 第 45 行：缺少 null 检查，可能导致空指针异常

🟡 建议改进
- 函数 processData() 过长，建议拆分为多个小函数

🟢 可选优化
- 变量名 x 可以改为更具体的名称如 userCount
```
